# Отчёт о домашней работе 6 "Бэкапы"
При работе над ДЗ был подготовлен bash скрипт [prep.sh](./prep.sh). Скрипт проводит все подготовительные работы:
1. Создаёт инфраструктуру для теста в Yandex Cloud: сеть, подсеть, 8 vm/
2. Устанавливает на vm требуемые пакеты
    * на 2 vm устанавливается Postgresql 18 и WAL-G
    * на vm-otus1 1 устанавливается Python venv и скрипт app2.py имитирующий работу приложения и добавляющий строки в базу данных 
    * vm-otus1 считается "рабочей", vm-otus2 - тестовая система для проверки восстановления из резервной копии 
3. После запуска Postgres vm-otus1 снимается первая полная резервная копия wal-g
4. vm-otus1 загружается дамп БД demo (https://postgrespro.ru/education/demodb). 
5. vm-otus1 снимается вторая полная резервная копия wal-g. Проверяется количество строк в таблице booking
6. Файлы резервной копии копируются на vm-otus2.  Т.к добавление данных в это время не происходит, резервная копия - это вторая полная копия БД на vm-otus1
7. vm-otus2 БД восстанавливается из резервной копии. Проверяется количество строк в таблице booking и сравнивается с vm-otus1   
8. vm-otus1 Запускает приложение app2.py добавляющее строки в таблицы. Периодичность вставки (в приложенных примерах 0.3сек и 0.1сек) и время работы регулируется  
9. Через заданное количество секунд (в приложенных примерах 30сек и 600сек) сервис Postgres на vm-otus1 останавливается, имитируя аварию на сервере 
10. Файлы резервной копии повторно копируются на vm-otus2.  Т.к после начала вставки данных полные резервные копии не создавались, резервная копия - это вторая полная копия БД на vm-otus1 + дельты из wal файлов.
11. vm-otus2 БД восстанавливается из резервной копии. Проверяется количество строк в таблице booking
12. vm-otus1 запускается Postgres. Проверяется количество строк в таблице booking и сравнивается с vm-otus2. 
Т.к резервная копия из которой востановлена БД vm-otus2 это не "полная" копия снятая командой `wal-g backup-push` количество строк в БД vm-otus2 может быть меньше,чем в БД vm-otus1


Скрипт многократно запускался командой
```bash
./prep.sh | tee out_$(date +%F_%T).txt
```
 с проверкой результатов.  
 Вывод сохранялся в файлы, примеры
 * [out_2026-04-06_17-12-35.txt](./out_2026-04-06_17-12-35.txt) -  время работы приложения app2.py - 30сек c записью каждые 0.3 сек 
 * [out_2026-04-06_17-28-46.txt](./out_2026-04-06_17-28-46.txt) - время работы приложения app2.py 600 сек с записью каждые 0.1 сек.


## Пояснения к файлу [out_2026-04-06_17-28-46.txt](./out_2026-04-06_17-28-46.txt) с результатами работы скрипта 
Логические этапы подготовки и проверки разделены строками. Для длительных этапов приведено время окончания этапа и для сравнения время начала работы скрипта

1. Подготовка инфраструктуры (сеть, 2 vm) установка программ (Postgers, Python venv) до записи

Важно, в конфигурацию Postgres добавляется файл с параметрами архивации - `/etc/postgresql/18/main/conf.d/wal-g.conf` 
```ini
wal_level=replica
archive_mode=on
archive_timeout=60
archive_command = '/usr/local/bin/wal-g wal-push %p'
restore_command = '/usr/local/bin/wal-g wal-fetch %f %p'
```
Параметры работы wal-g хранятся в файле с размещением по умолчанию `/var/lib/postgresql/.walg.json`
```json
{
  "WALG_FILE_PREFIX": "/mnt/backup",
  "WALG_COMPRESSION_METHOD": "brotli",
  "WALG_DELTA_MAX_STEPS": "5",
  "WALG_UPLOAD_DISK_CONCURRENCY": "4",
  "PGDATA": "/var/lib/postgresql/18/main",
  "PGHOST": "/var/run/postgresql"
}
```

2. `vm-otus1 Создание первой резевной копии WAL-G` - Первая полная резервная копия созадаётся после старта Postgres, до начала фактического использования. 
`sudo -u postgres /usr/local/bin/wal-g backup-push /var/lib/postgresql/18/main ` - создать полную резервную копию
Отображается список доступных резервных копий
3. `Cоздаём БД Demo и заполняем данными` - в Postgres vm-otus1 загружается дамп БД demo PostgresPro Air 
После загрузки проверяется количество строк в таблице booking - 1292893
4. `vm-otus1 Создание второй резевной копии WAL-G` - создаётся резервная копия и отображается список доступных, в котором теперь две копии  
5. `Переносим файлы резервной копии на VM vm-otus2 и проверяем восстановление` Файлы копируются командой rsync по ssh между vm-otus1 -> vm-otus2. Для этого предварительно на vm-otus1 создаётся ключ 
6. `vm-otus2 Запускаем скрипт восстановления из резервной копиии (до точки LATEST):` на vm-otus2 развёртывается резервная копия из перенесённых файлов.  
Очищается каталог данных и запускается развёртывание копии`sudo -u postgres  /usr/local/bin/wal-g backup-fetch /var/lib/postgresql/18/main LATEST`  
Создаётся файл recovery.signal и запускается сервис Postgres   
7. `vm-otus2 Запрашиваются данные из таблицы БД.` - 1292893
Сравниваем количество строк в таблице bookings на серверах vm-otus2=1292893 и vm-otus1=1292893 - Одинаковое
8. `vm-otus1 Запускаем приложение, которое каждые 0.1 секунду добавляет запись в таблицу booking`
через 600 сек принудительно останавливаем сервис Postgres `vm-otus1 Останавливается Postgres для имитации аварийной остановки`  
Выведена информация о работе приложения (сокращённая и редактированная). Количество строк в таблице booking 1298885, но проверка сделана до отключения сервера.
9. `vm-otus1 Повторно копируем каталог backup на тестовый сервер vm-otus2`
10. `vm-otus2 Запускаем скрипт восстановления из резервной копиии (до точки LATEST)` методика аналогичная п. 6
11. `vm-otus2 Запрашиваются данные из таблицы БД. ` - таблица booking - 1298907 строк
12. Сравнения данных восстановленных из резервной копии с оригиналом, `vm-otus1 стартовал кластер Postgres main` + `vm-otus1 Запрашиваются данные из таблицы БД.`
13. Сравниваем количество строк в таблице bookings на серверах после второго теста vm-otus2 = 1298907 и vm-otus1 = 1298907

**Итог**: Резервная копия WAL-G позволяет восстановиться на момент остановки сервиса Postgres без потери данных. 
Возможно, в условиях реальных аварий, например, при отключении электричества, данные резервной копии будут отставать от данных БД на некоторое количество транзакций.
В условиях тестирования резервная копия хранилась на файловой системе виртуальной машины (но, на отдельном диске), поэтому при отключении электричества или крахе vm потребуется заметное время для доступа к файлам резервных копий. Поэтому рекомендуется, хранить файлы резервной копии на внешнем хранилище, например, типа S3 или shared smb.
