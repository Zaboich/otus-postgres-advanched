# Отчёт о домашней работе 5 "Высокая доступность: развертывание Patroni"
При работе над ДЗ был подготовлен bash скрипт [prep.sh](./prep.sh). Скрипт проводит все подготовительные работы:
1. Создаёт инфраструктуру для теста в Yandex Cloud: сеть, подсеть, 8 vm/
2. Устанавливает на vm машинные требуемые пакеты
    на 3 vm устанавливается ETCD и настраивает кластер etcd
    на 3 vm устанавливается Postgresql 18 и Patroni. Настраивается и запускается кластер Patroni
    на 1 vm устанавливается и настраивается балансировщик Haproxy для балансировки запросов от клиентов
    на 1 vm устанавливается клиент Postgres psql. Эта vm имитирует обращение клиентов к БД
3. После настройки и запуска кластера Patroni с тестовой машины в БД загружается дамп БД достаточно большого объёма 
4. Проверяется доступность данных из созданной БД demo. 
5. Останавливается сервис patroni на vm лидере кластера 
6. Проверяется состояние кластера. 
7. Повторно проверяется доступность данных в БД demo. Данные должны быть доступны благодаря автоматическому failover перестроению кластера patroni и обновлению данных о новом лидере кластера в Haproxy  
8. Запускает сервис patroni на бывшем лидере patroni. 
9. Проверяется состояние кластера. Бывший лидер должен подключится к кластеру и получить роль реплики.
10. Проводится ручное переключение лидера. Роль лидера передаётся vm изначально (при старте кластера) ставшей лидером. 
    Для этого определяется текущий лидер и выполняется команда patronictl switchover
11. Проверяется доступность данных после ручного переключения лидера


Скрипт многократно запускался командой
```bash
./prep.sh | tee out_$(date +%F_%T).txt
```
 с проверкой результатов. Вывод сохранялся в файлы, например [out_2026-03-24_18-52-28.txt](./out_2026-03-24_18-52-28.txt) [out_2026-03-25_10-45-31.txt](./out_2026-03-25_10-45-31.txt).

Предварительный этап - **подготовка инфраструктуры** в Yandex Cloud с помощью cli инструментов `yc`
Этап **подготовки кластера ETCD**. Запускается скриптом [install_etcd.sh](./install_etcd.sh) На vm 1-3 устанавливается пакет etcd и задаётся конфигурация кластера. 
Важный момент - конфигурация задаётся как набор env переменных в `/etc/default/etcd/`. Возможны другие варианты сохранения конфигурации.
После подготовки конфигурации сервис etcd остаётся остановленным.
Этап **запуск кластера ETCD**. На всех 3-х vm запускается команда 
```
nohup sudo systemctl start etcd  > /dev/null 2>&1 &
```
Важно: На всех машинах выполняется практически одновременно без ожидания результата выполнения. Это требуются для успешного согласования кластера при заданной конфигурации.
После запуска ETCD выжидается пауза 20 сек до окончания инициализации кластера.

Этап **проверка состояния кластера ETCD**. Состояние кластера проверяется на каждой vm командой `etcdctl endpoint status --cluster -w table`. Вывод в виде таблицы.

Этап **установка Postgres**. Запускается скрипт [install_postgresql.sh](./install_postgresql.sh) на vm 4-6, входящих в кластер Patroni. На vm 8, которая будет использоваться для имитации клиента БД, запускается скрипт [install_psql-client.sh](./install_psql-client.sh) устанавливающий клиент psql.

Этап **установка Patroni**. На vm 4-6 запускается скрипт [install_patroni.sh](./install_patroni.sh). Скрипт устанавливает Python 3.12, формирует виртуальное окружение venv и устанавливает pip пакет patroni. В файл шаблона конфигурации [patroni.yml](./patroni.yml) загружается на каждую vm и в нём устанавливаются текущие значения соответствующие. Запуск Patroni оформляем в виде сервиса systemd.

Этап **запуск ноды - лидера кластера Patroni**. [init_patroni_leader.sh](./init_patroni_leader.sh). Останавливается сервис Postgres, очищается data директория. Стартует сервис Patroni, который инициализирует новую БД Postgres и кластер Patroni. Нода Patroni становится лидером кластера.
После паузы, для инициализации, проверяется состояние кластера на одной ноде `/opt/patroni/bin/patronictl -c /etc/patroni/patroni.yml list`.
Есть вариант инициализации лидера кластера с уже работающей и заполненной БД [../hw5.1/init_patroni_leader.sh](../hw5.1/init_patroni_leader.sh)

Этап **запуск ноды Patroni на репликах**. [init_patroni_replica.sh](./init_patroni_replica.sh). Происходит аналогично запуску лидера.

Этап **проверка состояния кластера Patroni**. Состояние кластера проверяется на каждой vm 4-6 командой `/opt/patroni/bin/patronictl -c /etc/patroni/patroni.yml list`.

Этап **установка и запуск балансировщика Haproxy**. [install_haproxy.sh](./install_haproxy.sh) Устанавливается Haproxy и устанавливается конфигурация на основе шаблона [haproxy.cfg](./haproxy.cfg)

На этом подготовка и запуск кластера завершены. Начинается использование/тестирование.

Этап **загрузка данных в БД**. На тестовую vm 8 скачивается дамп БД demo, используемый для различных экспериментов с Postgres. И дамп загружается в БД. 
Важно: Здесь и далее все подключения от psql на тестовой vm (клиента) **к БД происходят через Haproxy**.
После окончания загрузки дампа сразу проверяется состояние кластера с каждой vm.

Этап **проверки доступности БД с тестовой VM**. Выполняется запрос `SELECT * FROM airplanes_data`, выводятся данные таблицы.

Этап **Failover**. Через http запрос `http://{ADDR_VM}:8008/cluster` определяется лидер кластера (это должна быть vm 4, но кластер мог перестроиться). На полученной vm выполняется принудительная остановка Patroni `sudo systemctl stop patroni` .
Важно. После Failover требуется время на выбор нового лидера и перестроение кластера. Ожидание 5 сек.

Определяем **новый Patroni cluster Leader** http `http://{ADDR_VM}:8008/cluster` запросом к работающей vm

Ожидание 30 сек для **обновления данных о новом лидере в Haproxy**. У Haproxy есть задержка при определении нового лидера.

Этап **проверки доступности БД с тестовой VM**. После отключения первоначального лидера, повторно выполняется запрос `SELECT * FROM airplanes_data`, выводятся данные таблицы. **Данные аналогичны первому запросу**.

Этап **Запуск остановленного Patroni на vm 4**. На vm 4 выполняется `sudo systemctl start patroni`.

**Проверка состояния кластера после запуска Patroni на vm 4**. Видно, что нода получила роль Replica.

Этап **Switchover с первоначальным Leader** vm 4. На vm 4 выполняется команда `/opt/patroni/bin/patronictl -c /etc/patroni/patroni.yml switchover --leader vm-otus5 --candidate vm-otus4 --force`
Команда выводит состояния кластера до и после операции. Новым лидером становится vm 4

**Проверка состояния кластера после switchover** на каждой vm 4-6

Ожидание 30 сек **для обновления данных о лидере кластера в Haproxy** после switchover

Этап **проверки доступности БД с тестовой VM**. Выполняется запрос `SELECT * FROM airplanes_data`, выводятся данные таблицы. Данные аналогичны первому и второму запросу.

Все тесты пройдены. Длительность выполнения всех тестов около 10 мин.
